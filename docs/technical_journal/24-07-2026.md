### Error 2: settings file — wrong extension and wrong directory

- Next run failed with:
  ```
  NoMethodError: undefined method '[]' for nil
  lib/boukensha/tasks/base.rb:39:in 'Boukensha::Tasks::Base.fetch'
  lib/boukensha/tasks/base.rb:9:in 'Boukensha::Tasks::Base.provider'
  examples/example.rb:16
  ```
- Root cause, two compounding mistakes in how a first `.boukensha/` config dir had been created:
  - **Wrong extension** — `Boukensha::Config#load_settings` only looks for `settings.yaml`; the file had been saved as `settings.yml`.
  - **Wrong directory** — `examples/example.rb` sets `BOUKENSHA_DIR` to a path computed relative to itself (`File.expand_path("../../../.boukensha", __dir__)`), which resolves to `week1_baseline/.boukensha` (three levels up from `examples/`). The file had been placed at the outer repo root instead.
  - With no `settings.yaml` found at the resolved dir, `Config` silently fell back to `{}`. `config.tasks(:player)` then returned `nil`, and `Tasks::Base.provider(nil)` crashed on `nil["provider"]` — a config lookup failure disguised as a Ruby `NoMethodError`, unhelpful for someone who doesn't read Ruby yet.
- Fix:
  - Renamed `settings.yml` → `settings.yaml`.
  - Added one more `../` to `example.rb`'s `BOUKENSHA_DIR` computation (3 levels → 4), so it resolves to the actual repo root (`claude-code-camp-2026-Q2/.boukensha`) — matching the path the README's own worked example shows (`/home/andrew/Sites/Claude-Code-Camp/.boukensha`).
  - Added `.boukensha/.env` (holds `ANTHROPIC_API_KEY`) and `.boukensha/prompts/player/system.md` (the per-task system-prompt override referenced by `prompt_override.system: true` in `settings.yaml`).
- `./week1_baseline/bin/00_config` now runs clean end-to-end and matches the README's expected output (`Tasks: player`, `Provider: anthropic`, `Prompt override? true`, `API key set? true`).
